How to Verify This Site

Anyone can clone a website. Here is how to prove you are reading the real one.

The problem

A state actor or hostile group could copy this entire site, replace the manifesto with propaganda, and run the same "verified" badge infrastructure over their own signing keys. A green badge alone is not enough — you must also check whose keys signed it.

The official signing key fingerprints

Every page on this site shows a badge in the footer. When the badge is green, it displays one or more short key fingerprints — the first 8 and last 4 characters of each Ed25519 public key that signed this version. For example: 

✓ Signed · bafybeic… · v7 · c1688ff0…b5c3

The current official key fingerprints are published here and must be cross-checked against at least one independent source before you trust this site. A fake site would show different fingerprints.

How to cross-check the fingerprints

Verify the fingerprints above match what you find on at least one of these independent channels:

  • Nostr — search for #cjp-keys events from the party's official Nostr account. The full public key will be in the event body.
  • Physical materials — the fingerprint is printed on CJP flyers distributed at protests and university campuses. If you received a flyer, check the bottom corner.
  • Trusted contacts — ask someone who was already following CJP before any censorship. If the fingerprint they show you matches the badge on this page, you are on the real site.
  • GitHub source repo — compare the CID shown in the badge against latest.json on GitHub. The cid field must be identical. The signer or signatures[].signer field must match the fingerprint in the badge.
  • IPFS directly — the signed CID in the badge links directly to content on IPFS. Even if every clearweb domain is seized, the content at that CID is immutable and signed.

What the badge actually checks

  1. Fetches latest.json from GitHub — a pointer to the current IPFS CID.
  2. Verifies the Ed25519 signature(s) against keys hardcoded inside this very file (not fetched from GitHub — so GitHub cannot be used to inject fake keys).
  3. Fetches integrity.json from IPFS via the signed CID. Because IPFS is content-addressed, this file cannot be tampered with without invalidating the CID.
  4. Hashes this page's source code and compares it to the hash in integrity.json. A mirror serving modified content will fail this check even if the CID signature is valid.

A site that fails any of these steps shows a red badge. A site that cannot reach GitHub or IPFS shows a grey badge — this does not mean the site is fake, only that live verification is unavailable.

Extending trust to new party members

The threshold is currently 1-of-1. As more members join, the threshold should be raised so no single compromised key can publish a rogue update. The workflow to add a new signer:

  1. New member runs publisher keygen --out ~/.cjp/ and shares their public key hex.
  2. Add the new pubkey to HARDCODED_SIGNERS in packages/site/js/verify.js and to trusted-signers.json. Raise HARDCODED_THRESHOLD and trusted-signers.json#threshold as appropriate.
  3. Rebuild: node scripts/build.js
  4. Compute new CID: ipfs add -r --cid-version=1 --quieter dist/
  5. Each existing signer runs publisher sign --cid <new-cid> — once threshold signatures are collected, publisher publish broadcasts the update.
  6. Commit latest.json and trusted-signers.json to the repo.

Until a new CID (containing the updated verify.js) is signed and deployed, the new key has no effect. This prevents a rogue actor from quietly adding themselves to the signers list.

Manual update — when GitHub is unreachable

If GitHub and every Nostr relay you can reach are down or blocked, the developer can still publish a signed update through any other channel — a friend's tweet, a Signal group, a printed flyer, a QR code. The trust anchor is the Ed25519 signature, not the transport. Paste the signed latest.json below; your browser verifies it against the hardcoded signer keys before storing it locally.

Verified updates are stored in this browser's localStorage. Clearing site data or browser cache removes them. The override is used in addition to GitHub and Nostr — whichever source has the highest signed version wins.

Peer attestation network

Every mirror publishes a periodic attestation — a signed Nostr event listing every other mirror it has observed via heartbeats, plus the event IDs of every other mirror's attestation it has ingested. Two endorsement layers in one event:

  • Heartbeat-based: "I have personally seen this peer broadcasting."
  • Attestation-of-attestation: "I have observed this peer's own attestation event existing at this time" — Nostr event IDs are content-addressed, so referencing one cryptographically requires the referenced event to predate yours. Back-dating an attestation chain requires forging every event that references it.

The trust score below counts distinct attesting mirrors across both layers. A peer endorsed by ≥ 2 long-standing mirrors is credible; a peer with attesters = 1 is either new or only self-observed — verify out-of-band. The cross-attestation graph visualises the clique; Sybil clusters that only attest to each other appear as a dense isolated subgraph.

Unencrypted by design — the only way "this fake was never one of us" can be verified by third parties is if anyone can read the attestations. Trust is public, queryable, reproducible.

What this cannot protect against

If an attacker obtains the actual CJP signing private keys and takes over the GitHub repository, they could publish a modified site with a valid green badge. This is why key custody is distributed across multiple independent party members: a valid update requires signatures from a threshold of keyholders.

No software mechanism can fully substitute for a community that knows and trusts its own members. The badge is a tool, not a guarantee.
